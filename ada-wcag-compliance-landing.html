Legal Framework

What Is ADA Compliance for Websites?

The Americans with Disabilities Act (ADA) was signed into law in 1990. Courts have consistently extended its requirements to websites and digital products.

Title III of the ADA prohibits discrimination based on disability in places of public accommodation. The Department of Justice has affirmed that websites qualify as places of public accommodation under the ADA.

While the ADA itself doesn't specify a technical standard, courts and the DOJ widely reference WCAG 2.1 Level AA as the benchmark for what constitutes an accessible website.

Any business with a website that serves the public — whether for e-commerce, services, or information — is exposed to ADA litigation if their site is inaccessible to people with disabilities such as blindness, low vision, deafness, motor impairments, or cognitive disabilities.

The Section 508 amendment further requires federal agencies and organizations receiving federal funding to meet WCAG 2.0 AA at minimum for all digital content.

Federal Lawsuits

Plaintiffs can sue in federal court under the ADA. Settlements often reach $25,000–$100,000+, plus attorney fees and mandatory remediation costs.

Demand Letters

Serial litigants send demand letters requiring immediate remediation. Ignoring them escalates to costly litigation.

Reputational Damage

Public lawsuits damage brand reputation and signal that your company doesn't value inclusivity or equal access.

Government Contracts at Risk

Federal contractors and grant recipients must comply with Section 508 or risk losing funding and contract eligibility.

The Standard

WCAG 2.1 — Web Content Accessibility Guidelines

Published by the W3C, WCAG 2.1 is the internationally recognized technical standard for web accessibility. It is organized into three conformance levels.

Level A

Minimum

The bare-minimum baseline. Removing these barriers is essential but rarely sufficient. Failing Level A blocks entire user groups from accessing your content.

Level AA

Standard

The widely-adopted legal standard. WCAG 2.1 AA covers the most significant accessibility barriers. Required by the DOJ for public-facing websites and Section 508 compliance.

Level AAA

Enhanced

The highest standard, often applied to specialized contexts (government, healthcare, education). Not required for all content but represents best practice for inclusion.

The Four Principles

POUR — The Foundation of WCAG

Every WCAG guideline is organized under four core principles. Content must be Perceivable, Operable, Understandable, and Robust to be considered accessible.

P — Perceivable

Users Must Be Able to Perceive All Information

Information and user interface components must be presentable to users in ways they can perceive. Content cannot be invisible to all of a user's senses — users who are blind, deaf, or have low vision rely on alternative presentations of content.

All non-text content (images, icons, charts) must have descriptive alt text
Pre-recorded audio must have a text transcript
Pre-recorded video must have synchronized captions
Live video streams must include live captions
Color must not be the only visual means of conveying information
Text must have a contrast ratio of at least 4.5:1 against its background
Large text (18pt+) must meet a 3:1 contrast ratio
Text must be resizable up to 200% without loss of content
Images of text must be avoided; use real text instead
Content must be adaptable — presentable in simpler form without losing meaning
No auto-playing audio longer than 3 seconds without user control
Layout must use proper semantic structure (headings, lists, tables)
O — Operable

Users Must Be Able to Operate the Interface

User interface components and navigation must be operable by everyone. Users with motor disabilities, tremors, or those unable to use a mouse rely on keyboards, switch controls, and other assistive input devices to interact with interfaces.

All functionality must be accessible via keyboard alone (no mouse required)
No keyboard traps — users must be able to navigate away from any component
A skip navigation link must be provided to bypass repetitive content blocks
All pages must have descriptive titles identifying their purpose
Focus order must follow a logical, meaningful sequence
The purpose of each link must be clear from its link text alone
Timed content must offer options to pause, stop, or extend time limits
Moving or scrolling content must be able to be paused or stopped
Nothing must flash more than 3 times per second (seizure risk)
Multiple navigation methods must be available (search, sitemap, breadcrumbs)
Visible keyboard focus indicator must be present on all interactive elements
Touch targets must be at least 44×44 CSS pixels in size
U — Understandable

Users Must Be Able to Understand Content and UI

Information and the operation of the user interface must be understandable to all users. Cognitive disabilities, learning differences, and language barriers all affect whether a user can comprehend your content and predict how your interface will behave.

The language of each page must be identified in the HTML lang attribute
Language changes within the page must use the lang attribute on the element
No unexpected context changes when a component receives focus
Changing a setting must not automatically trigger a context change without warning
Navigation must be consistent across all pages of the site
Components with the same function must be labeled consistently
Input fields must have visible, descriptive labels — not placeholder text alone
Input errors must be clearly identified and described in text
Labels or instructions must be provided when content requires user input
Important forms must allow users to review, correct, and confirm submissions
Error suggestion messages must be provided if input format is known
Unusual words, jargon, and abbreviations must have definitions or expansions
R — Robust

Content Must Work With Current and Future Assistive Technologies

Content must be robust enough that it can be reliably interpreted by a wide variety of user agents, including assistive technologies like screen readers, braille displays, and voice control software. As technologies evolve, your content should remain accessible.

HTML must be valid and well-formed — no duplicate IDs, properly nested tags
All UI components must expose their name, role, and value programmatically
Custom components must use appropriate ARIA roles and attributes
ARIA labels must be used on icon-only buttons and links
Interactive elements must have meaningful accessible names
Status messages (success/error) must be conveyed to screen readers via ARIA live regions
Dynamic content changes via JavaScript must be announced to assistive tech
Modal dialogs must trap focus and announce their role to screen readers
Form elements must be programmatically associated with their labels via for/id
Tables must use <th> with scope attributes for headers
Avoid using ARIA to override native HTML semantics unnecessarily
UI component state changes (expanded, selected, checked) must be programmatically exposed
What's Broken

Most Common Accessibility Failures

The WebAIM Million annual report consistently finds the same failures across 96% of the top one million websites.

Perceivable

Low Color Contrast

Light gray text on white backgrounds fails the 4.5:1 contrast ratio. Found on 83% of home pages surveyed.

Fix: Use a contrast checker tool. Ensure body text meets 4.5:1 and large text meets 3:1 against its background color.
Perceivable

Missing Alt Text

Images without alt attributes are invisible to screen reader users. Found on 55% of home pages — the most common failure type.

Fix: Add descriptive alt attributes to all meaningful images. Use alt="" for decorative images only.
Robust

Empty or Missing Form Labels

Input fields without associated labels leave screen reader users unable to determine what to enter. Found on 46% of home pages.

Fix: Use <label for="id"> elements associated with every form input. Never rely on placeholder text as a label.
Operable

Missing or Invisible Focus Indicators

Removing the browser's default outline via CSS makes keyboard navigation impossible for sighted keyboard users.

Fix: Never use outline: none without providing a custom, clearly visible focus style. Use :focus-visible for a clean approach.
Understandable

Ambiguous Link Text

"Click here," "read more," and "learn more" links provide no context out of the page flow — a critical barrier for screen reader users navigating by links.

Fix: Write descriptive link text: "Read our accessibility guide" instead of "read more." Use aria-label if the surrounding context cannot be changed.
Robust

Missing Page Language

Without a lang attribute on the <html> element, screen readers may use the wrong language voice profile, mangling pronunciation entirely.

Fix: Always declare <html lang="en"> (or the appropriate language code). For multilingual pages, add lang attributes to individual sections.
